# 1.1 클라우드 아키텍처(Cloud Architecture)란?

먼저 "아키텍처(Architecture)"라는 단어부터 이해해 보겠습니다.

아키텍처란 **시스템을 어떻게 구성하고 설계할 것인지에 대한 전체적인 구조와 설계 방식**을 의미합니다.

예를 들어 집을 짓는다고 생각해 보겠습니다.

집을 지을 때는 바로 벽돌을 쌓지 않습니다.

먼저 다음과 같은 설계를 합니다.

* 방은 몇 개로 만들 것인가?
* 화장실은 어디에 배치할 것인가?
* 전기와 수도는 어떻게 연결할 것인가?
* 지진이나 태풍에도 안전한 구조인가?

이러한 설계도를 **건축 아키텍처(Architecture)**라고 합니다.

IT 시스템도 마찬가지입니다.

웹사이트나 모바일 앱, 쇼핑몰, 게임 서버 등을 구축할 때도 단순히 서버를 여러 대 설치하는 것이 아니라, 전체 시스템을 어떻게 구성할 것인지 먼저 설계해야 합니다.

이를 **클라우드 아키텍처(Cloud Architecture)**라고 합니다.

예를 들어 온라인 쇼핑몰을 AWS에 구축한다고 가정해 보겠습니다.

```text
                           사용자(User)
                                │
                             Internet
                                │
                         Elastic Load Balancer
                                │
          ┌──────────┴──────────┐
          │                                          │
       EC2(Web1)                                 EC2(Web2)
          │                                          │
          └──────────┬──────────┘
                                │
                            Amazon RDS
                                │
                             Amazon S3
```

이 그림은 하나의 간단한 클라우드 아키텍처입니다.

각 서비스는 다음과 같은 역할을 합니다.

| 구성 요소     | 역할                           |
| ------------- | ------------------------------ |
| Load Balancer | 사용자 요청을 여러 서버로 분산 |
| EC2           | 웹 애플리케이션 실행           |
| Amazon RDS    | 데이터베이스 저장              |
| Amazon S3     | 이미지 및 파일 저장            |

즉, 아키텍처는 **서비스를 어떤 구조로 배치할 것인지 설계하는 것**입니다.

---

# 1.2 왜 좋은 아키텍처가 필요한가?

클라우드에서는 서버를 쉽게 만들 수 있습니다.

그러나 서버를 많이 만든다고 해서 좋은 시스템이 되는 것은 아닙니다.

다음 두 가지 사례를 비교해 보겠습니다.

## 사례 1. 잘못된 설계

한 스타트업이 EC2 인스턴스 한 대에 모든 서비스를 설치했습니다.

```text
EC2

├─ Web Server
├─ Database
├─ File Storage
├─ Application
└─ Cache
```

처음에는 문제가 없었습니다.

하지만 사용자가 늘어나자 다음과 같은 문제가 발생했습니다.

* 서버 속도가 느려짐
* 메모리 부족
* CPU 사용률 100%
* 서버 장애 시 전체 서비스 중단
* 백업 어려움
* 유지보수 어려움

---

## 사례 2. AWS 권장 설계

AWS에서는 역할을 분리하여 설계하는 것을 권장합니다.

```text
                Load Balancer
                      │
         ┌────────────┴────────────┐
         │                         │
      EC2(Web1)               EC2(Web2)
         │                         │
         └────────────┬────────────┘
                      │
                 Amazon RDS
                      │
                  Amazon S3
```

이 구조에서는 다음과 같은 장점이 있습니다.

* 서버 하나가 고장 나도 서비스 지속
* 데이터베이스는 별도로 관리
* 파일 저장소도 분리
* 서버를 쉽게 추가 가능
* 유지보수 용이

이처럼 **올바른 아키텍처는 서비스의 안정성과 확장성을 크게 향상시킵니다.**

---

# 1.3 AWS는 왜 Well-Architected Framework를 만들었을까?

AWS는 전 세계 수백만 고객의 시스템을 운영하면서 다양한 성공 사례와 실패 사례를 분석했습니다.

예를 들어 다음과 같은 공통적인 문제를 발견했습니다.

* 서버를 하나만 구성하여 장애 발생 시 서비스가 중단됨
* 백업이 없어 데이터가 손실됨
* 보안 설정이 미흡하여 정보가 유출됨
* 사용하지 않는 리소스를 계속 운영하여 비용이 증가함

이러한 문제를 줄이기 위해 AWS는 **모범 사례(Best Practices)**를 정리하였습니다.

이것이 바로 **AWS Well-Architected Framework**입니다.

즉, Well-Architected Framework는 "AWS가 권장하는 좋은 클라우드 설계 기준"이라고 이해하면 됩니다.

---

# 1.4 AWS Well-Architected Framework란?

AWS Well-Architected Framework는 **안전하고, 효율적이며, 안정적인 클라우드 시스템을 설계하기 위한 모범 사례 집합**입니다.

쉽게 말하면, 건축에서 건축법과 설계 기준이 있듯이, AWS에서는 클라우드 시스템을 설계할 때 지켜야 할 원칙을 정리한 것입니다.

AWS는 다음과 같이 정의합니다.

> "클라우드 아키텍처를 평가하고 개선하기 위한 모범 사례 프레임워크"

이 프레임워크를 활용하면 현재 구축한 시스템을 점검하고, 부족한 부분을 개선할 수 있습니다.

---

# 1.5 Well-Architected Framework의 핵심 목표

AWS는 다음과 같은 목표를 달성하기 위해 Well-Architected Framework를 제공합니다.

## ① 안정성(Reliability)

서비스가 장애 상황에서도 계속 동작해야 합니다.

예를 들어 은행 시스템은 24시간 운영되어야 합니다.

서버 한 대가 고장 났다고 모든 서비스가 중단되어서는 안 됩니다.

---

## ② 보안(Security)

클라우드에는 중요한 고객 정보가 저장됩니다.

따라서 다음과 같은 보안이 필요합니다.

* 접근 권한 관리
* 데이터 암호화
* 사용자 인증
* 로그 기록

---

## ③ 성능(Performance)

사용자가 많아져도 서비스가 느려지지 않아야 합니다.

예를 들어 콘서트 티켓 예매 사이트는 특정 시간에 수십만 명이 동시에 접속할 수 있습니다.

이때도 빠른 응답 속도를 유지해야 합니다.

---

## ④ 비용 절감(Cost Optimization)

클라우드는 사용한 만큼 비용을 지불합니다.

따라서 필요 이상의 리소스를 사용하면 비용이 증가합니다.

AWS는 적절한 리소스를 선택하고 자동으로 확장하거나 축소하는 설계를 권장합니다.

---

## ⑤ 운영 자동화(Operational Excellence)

사람이 반복적으로 수행하는 작업은 자동화하는 것이 좋습니다.

예를 들어 서버 생성, 배포, 백업 등을 자동화하면 오류를 줄이고 운영 효율을 높일 수 있습니다.

---

## ⑥ 지속 가능성(Sustainability)

최근에는 친환경 IT가 중요해졌습니다.

필요 이상의 서버를 운영하면 전력 소비가 증가하고 탄소 배출도 늘어납니다.

AWS는 효율적인 리소스 사용을 통해 환경 보호에도 기여할 수 있도록 설계를 권장합니다.

---

# 1.6 Well-Architected Framework의 6가지 Pillars

Well-Architected Framework는 6개의 핵심 설계 원칙(Pillar)으로 구성됩니다.

```text
              AWS Well-Architected Framework

                      [Cloud Architecture]

      ┌──────────────────────────────────────────┐
      │ 1. Operational Excellence                │
      │ 2. Security                              │
      │ 3. Reliability                           │
      │ 4. Performance Efficiency                │
      │ 5. Cost Optimization                     │
      │ 6. Sustainability                        │
      └──────────────────────────────────────────┘
```

각 Pillar의 의미를 간단히 살펴보겠습니다.

| Pillar                 | 핵심 질문                                | 쉽게 이해하기           |
| ---------------------- | ---------------------------------------- | ----------------------- |
| Operational Excellence | 운영을 효율적으로 하고 있는가?           | 자동화와 지속적인 개선  |
| Security               | 안전한 시스템인가?                       | 데이터와 시스템 보호    |
| Reliability            | 장애가 발생해도 계속 서비스할 수 있는가? | 안정성과 복구 능력      |
| Performance Efficiency | 필요한 성능을 효율적으로 제공하는가?     | 성능과 확장성           |
| Cost Optimization      | 비용을 낭비하지 않는가?                  | 비용 절감과 효율        |
| Sustainability         | 친환경적으로 운영하고 있는가?            | 에너지 효율과 탄소 절감 |

이 여섯 가지는 서로 독립적인 것이 아니라, 하나의 균형 잡힌 아키텍처를 구성하는 요소입니다.

---

# 1.7 Well-Architected Framework를 실제 프로젝트에서 어떻게 활용할까?

프로젝트를 진행할 때는 다음과 같은 질문을 스스로에게 던져야 합니다.

* 운영 자동화는 충분한가?
* 보안은 적절하게 구성되었는가?
* 장애가 발생해도 서비스를 계속 제공할 수 있는가?
* 성능은 충분한가?
* 비용은 효율적인가?
* 불필요한 리소스를 사용하고 있지는 않은가?

이러한 질문에 답하면서 시스템을 개선하는 과정이 바로 Well-Architected Framework를 활용하는 방법입니다.

# 1.9 핵심 정리

이 장에서는 AWS Well-Architected Framework의 기본 개념과 필요성을 학습했습니다.

핵심 내용을 다시 정리하면 다음과 같습니다.

* **클라우드 아키텍처**는 시스템을 효율적으로 설계하는 전체 구조를 의미합니다.
* **AWS Well-Architected Framework**는 AWS가 제시하는 클라우드 설계 모범 사례(Best Practice)입니다.
* 좋은 아키텍처는 안정성, 보안, 성능, 비용, 운영 효율, 지속 가능성을 모두 고려해야 합니다.

  1. Operational Excellence
  2. Security
  3. Reliability
  4. Performance Efficiency
  5. Cost Optimization
  6. Sustainability

* AWS CCP 시험에서는 **각 Pillar의 목적과 대표적인 AWS 서비스 및 적용 사례를 구분하는 문제가 자주 출제**됩니다.

# 2.2 Operational Excellence (운영 우수성)

## 2.2.1 Operational Excellence란?

Operational Excellence는 **운영을 효율적으로 수행하고 지속적으로 개선하는 능력**을 의미합니다.

AWS에서는 운영을 사람의 경험에만 의존하지 않고, 가능한 한 자동화하여 일관성과 안정성을 확보하는 것을 권장합니다.

즉,

> **"반복적인 작업은 자동화하고, 운영 과정은 지속적으로 개선하라."**

라는 것이 핵심입니다.

---

## 2.2.2 왜 필요한가?

예를 들어 회사에서 매주 새로운 버전의 프로그램을 배포한다고 가정해 보겠습니다.

### 수동 운영

```text
관리자 로그인
      ↓
파일 업로드
      ↓
서비스 중지
      ↓
프로그램 교체
      ↓
서비스 시작
```

문제점

* 작업 시간이 오래 걸림
* 관리자마다 작업 방식이 다름
* 실수 가능성 증가
* 야간 작업 필요

---

### 자동화 운영

```text
Git Push
     │
AWS CodePipeline
     │
자동 Build
     │
자동 Test
     │
자동 Deploy
```

장점

* 동일한 절차 반복
* 사람의 실수 감소
* 빠른 배포
* 장애 발생 시 빠른 복구

---

## 2.2.3 Operational Excellence의 핵심 원칙

AWS는 다음과 같은 운영 원칙을 제시합니다.

### ① 운영을 코드로 관리(Operations as Code)

서버를 직접 설정하는 대신 코드로 관리합니다.

예를 들어 **AWS CloudFormation**을 사용하면 동일한 인프라를 언제든지 다시 생성할 수 있습니다.

```text
CloudFormation Template

↓

EC2 생성

↓

VPC 생성

↓

RDS 생성

↓

자동 완료
```

**장점**

* 동일한 환경 구성
* 버전 관리 가능
* 빠른 복구

---

### ② 작은 변경을 자주 수행

한 번에 큰 변경을 하는 것보다 작은 변경을 자주 적용하면 문제가 발생했을 때 원인을 쉽게 찾을 수 있습니다.

---

### ③ 운영 절차를 자주 개선

운영 중 발견된 문제를 분석하여 지속적으로 개선합니다.

예를 들어 장애 발생 후 원인을 분석(Postmortem)하고 재발 방지 대책을 마련합니다.

| ------------------- | --------------------- |

---

## 2.2.5 실제 기업 사례

온라인 쇼핑몰을 운영하는 A사는 매주 금요일 밤마다 새로운 기능을 배포했습니다.

초기에는 개발자가 직접 서버에 접속하여 배포했지만, 배포 과정에서 파일 누락이나 설정 오류로 서비스 장애가 자주 발생했습니다.

AWS CodePipeline과 CloudFormation을 도입한 후에는 코드가 저장소에 반영되면 자동으로 빌드, 테스트, 배포가 이루어졌고, 배포 시간이 단축되었으며 장애도 크게 감소했습니다.

# 2.3 Security (보안)

## 2.3.1 Security란?

Security는 **데이터와 시스템을 보호하는 것**을 의미합니다.

AWS에서는 보안을 가장 중요한 요소 중 하나로 생각합니다.

보안은 단순히 해커를 막는 것이 아니라,

* 사용자 인증
* 접근 권한 관리
* 데이터 암호화
* 로그 기록
* 위협 탐지

등을 모두 포함하는 개념입니다.

---

## 2.3.2 공동 책임 모델(Shared Responsibility Model)

AWS 보안의 가장 중요한 개념은 **공동 책임 모델(Shared Responsibility Model)**입니다.

AWS와 고객이 각각 책임지는 영역이 다릅니다.

```text
          Shared Responsibility Model

      AWS 책임                고객 책임

 ─────────────────     ─────────────────────

 데이터센터              IAM 사용자 관리

 서버 하드웨어           비밀번호 관리

 네트워크                보안 그룹 설정

 전원                    데이터 암호화

 물리적 보안             운영체제 업데이트
```

### 쉽게 이해하기

아파트에 비유하면,

* 건물의 출입문과 CCTV는 관리사무소(AWS)가 관리합니다.
* 우리 집의 현관문 비밀번호와 귀중품 관리는 입주자(고객)가 책임집니다.

즉, AWS가 모든 보안을 대신해 주는 것이 아니라 고객도 자신의 영역을 책임져야 합니다.

---

## 2.3.3 보안의 핵심 원칙

### 최소 권한 원칙(Least Privilege)

사용자에게 필요한 권한만 부여합니다.

예를 들어,

* 개발자는 개발 서버만 접근
* 회계 담당자는 비용 정보만 조회
* 운영자는 서버 관리 권한

처럼 역할에 맞는 최소 권한을 부여해야 합니다.

---

### 데이터 암호화

데이터는 저장 중(At Rest)과 전송 중(In Transit) 모두 암호화하는 것이 좋습니다.

* 저장 데이터 → Amazon S3 암호화, Amazon EBS 암호화
* 전송 데이터 → HTTPS, TLS

---

### 로그 및 감사

누가 언제 어떤 작업을 했는지 기록하여 보안 사고를 추적할 수 있어야 합니다.

---

## 2.3.4 대표 AWS 서비스

| 서비스           | 역할                   |
| ---------------- | ---------------------- |
| AWS IAM          | 사용자 및 권한 관리    |
| AWS KMS          | 암호화 키 관리         |
| AWS Shield       | DDoS 공격 방어         |
| AWS WAF          | 웹 애플리케이션 방화벽 |
| Amazon GuardDuty | 위협 탐지              |
| AWS CloudTrail   | API 호출 및 감사 로그  |

---

## 2.3.5 실제 기업 사례

한 금융회사는 고객의 개인정보를 Amazon S3에 저장하고 있었습니다.

보안을 강화하기 위해 다음과 같은 조치를 수행했습니다.

* IAM 역할 기반 접근 제어
* S3 버킷 암호화
* CloudTrail을 통한 감사 로그 기록
* GuardDuty를 통한 이상 행위 탐지

그 결과 보안 수준이 크게 향상되고 규제 준수도 용이해졌습니다.

# 2.4 Reliability (신뢰성)

## 2.4.1 Reliability란?

Reliability는 **장애가 발생하더라도 서비스를 지속적으로 제공할 수 있는 능력**입니다.

즉,

> "시스템은 장애가 발생하는 것을 전제로 설계해야 한다."

라는 것이 AWS의 철학입니다.

---

## 2.4.2 왜 중요한가?

예를 들어 쇼핑몰에서 블랙프라이데이 세일이 진행되는 중 서버가 다운되면 어떻게 될까요?

* 주문 불가
* 고객 이탈
* 매출 손실
* 기업 이미지 하락

따라서 장애가 발생하더라도 서비스는 계속 운영되어야 합니다.

---

## 2.4.3 고가용성(High Availability)

AWS는 여러 개의 **가용 영역(Availability Zone, AZ)** 을 활용하여 서비스를 구성할 것을 권장합니다.

```text
             Elastic Load Balancer
                    │
         ┌──────────┴──────────┐
         │                     │
     AZ-1                    AZ-2
   EC2(Web1)              EC2(Web2)
         │                     │
         └──────────┬──────────┘
                    │
              Amazon RDS
              (Multi-AZ)
```

AZ-1에 장애가 발생하더라도 AZ-2가 서비스를 계속 제공합니다.

---

## 2.4.4 Reliability를 높이는 방법

### Auto Scaling

사용자가 증가하면 EC2를 자동으로 추가하고, 감소하면 자동으로 제거합니다.

### Elastic Load Balancing

사용자 요청을 여러 서버에 분산하여 특정 서버에 부하가 집중되지 않도록 합니다.

### 백업 및 복구

Amazon RDS 스냅샷, Amazon S3 버전 관리 등을 통해 데이터를 보호합니다.

### 장애 자동 복구

장애가 발생한 인스턴스를 자동으로 새 인스턴스로 교체합니다.

---

## 2.4.5 대표 AWS 서비스

| 서비스                 | 역할                       |
| ---------------------- | -------------------------- |
| Elastic Load Balancing | 트래픽 분산                |
| EC2 Auto Scaling       | 자동 확장 및 축소          |
| Amazon RDS Multi-AZ    | 데이터베이스 이중화        |
| Amazon Route 53        | DNS 및 장애 조치(Failover) |
| AWS Backup             | 중앙 집중식 백업           |

---

## 2.4.6 실제 기업 사례

동영상 스트리밍 서비스를 운영하는 B사는 한 리전에 두 개의 AZ를 사용하여 웹 서버를 배치했습니다.

평상시에는 두 AZ가 함께 서비스를 제공하고, 한 AZ에 장애가 발생하면 나머지 AZ가 서비스를 계속 처리합니다. 또한 Auto Scaling을 통해 사용자 증가 시 자동으로 서버를 추가하여 안정적인 서비스를 유지했습니다.

* **Operational Excellence**는 **운영 자동화와 지속적인 개선**에 초점을 둡니다.
* **Security**는 **데이터와 시스템을 안전하게 보호**하는 것이 목표이며, **Shared Responsibility Model**과 **최소 권한 원칙(Least Privilege)**이 핵심입니다.
* **Reliability**는 **장애를 전제로 설계**하여 서비스가 중단되지 않도록 하는 것이 목표이며, **Multi-AZ, Auto Scaling, ELB** 등의 서비스가 대표적인 구현 방법입니다.

> **암기 팁**
>
> * **Operational Excellence** → *운영을 더 잘하는 방법* (자동화)
> * **Security** → *안전하게 보호하는 방법* (권한·암호화)
> * **Reliability** → *계속 서비스하는 방법* (장애 대비)

이 세 가지 Pillar는 AWS Well-Architected Framework의 기반을 이루며, 실제 현업에서도 가장 먼저 고려하는 설계 원칙입니다. 다음 장에서는 **Performance Efficiency**, **Cost Optimization**, **Sustainability**를 학습하고, 여섯 개 Pillar가 어떻게 하나의 균형 잡힌 클라우드 아키텍처를 만드는지 살펴보겠습니다.

# 2.1 Performance Efficiency (성능 효율성)

## 2.1.1 Performance Efficiency란?

Performance Efficiency는 **필요한 성능을 가장 효율적인 방법으로 제공하는 능력**을 의미합니다.

많은 학생들이 "성능이 좋다"와 "성능 효율성이 좋다"를 같은 의미로 생각하지만, AWS에서는 다르게 봅니다.

예를 들어 CPU가 64개인 초고성능 서버를 사용하는 것은 성능은 뛰어나지만, 사용자가 10명뿐이라면 효율적이지 않습니다.

AWS에서는 다음과 같은 질문을 합니다.

> **"현재 서비스에 필요한 성능을 가장 효율적인 비용과 자원으로 제공하고 있는가?"**

즉, 성능만 높이는 것이 아니라 **필요한 만큼의 성능을 적절한 자원으로 제공하는 것**이 핵심입니다.

---

## 2.1.2 Performance Efficiency가 중요한 이유

온라인 쇼핑몰을 예로 들어보겠습니다.

평상시에는 하루 5,000명이 방문하지만,

블랙프라이데이에는 하루 50만 명이 접속할 수 있습니다.

### 기존(On-Premises)

```text
평상시

사용자 5,000명

↓

고성능 서버 10대 운영

↓

평소 대부분 놀고 있음
```

문제점

* 서버 자원이 낭비됨
* 유지비 증가
* 전력 소비 증가

---

### AWS 방식

```text
평상시

사용자 5,000명

↓

EC2 2대 운영

↓

블랙프라이데이

↓

EC2 20대로 자동 증가

↓

행사 종료

↓

다시 EC2 2대로 감소
```

이것이 AWS의 **Elasticity(탄력성)** 입니다.

필요할 때만 성능을 높이고, 필요 없으면 줄이는 것이 Performance Efficiency의 핵심입니다.

---

# 2.1.3 Performance Efficiency의 핵심 원칙

AWS는 다음과 같은 원칙을 제시합니다.

---

### ① 적절한 리소스 선택

모든 애플리케이션이 같은 서버를 사용할 필요는 없습니다.

예를 들어

| 서비스      | 적합한 리소스           |
| ----------- | ----------------------- |
| 웹 서버     | Amazon EC2              |
| 이벤트 처리 | AWS Lambda              |
| 컨테이너    | Amazon ECS / Amazon EKS |
| 데이터 분석 | Amazon EMR              |

적절한 서비스를 선택하는 것이 성능 향상의 첫걸음입니다.

---

### ② 최신 기술 적극 활용

AWS는 지속적으로 새로운 서비스를 제공합니다.

예를 들어

Intel CPU 대신

AWS Graviton Processor를 사용하면

* 성능 향상
* 비용 절감
* 전력 절감

을 동시에 얻을 수 있습니다.

---

### ③ 글로벌 서비스 활용

전 세계 사용자가 있다면

가까운 위치에서 서비스를 제공해야 합니다.

이를 위해

Amazon CloudFront를 사용합니다.

```text
한국 사용자

↓

서울 Edge Location

↓

CloudFront

↓

원본 서버
```

캐시를 이용하여 응답 속도를 크게 향상시킵니다.

---

### ④ Auto Scaling 활용

트래픽 변화에 따라

자동으로 서버를 증가 또는 감소시킵니다.

```text
사용자 증가

↓

Auto Scaling

↓

EC2 증가

↓

응답 속도 유지
```

| ----------------------- | ----------------- |

| Elastic Load Balancing  | 부하 분산         |
| Amazon CloudFront       | CDN 및 캐시       |
| AWS Lambda              | 서버리스 컴퓨팅   |

# 2.1.5 실제 기업 사례

# 2.2 Cost Optimization (비용 최적화)

## 2.2.1 Cost Optimization이란?

Cost Optimization은

**최소 비용으로 최대 효과를 얻도록 클라우드 자원을 사용하는 것**입니다.

AWS에서는

> "필요한 만큼만 사용하고,

필요하지 않으면 제거하라."

라는 철학을 가지고 있습니다.

---

## 2.2.2 왜 중요한가?

AWS는 사용한 만큼 과금(Pay-As-You-Go)합니다.

따라서

불필요한 리소스를 계속 실행하면

매달 비용이 증가합니다.

예를 들어

```text
개발 서버

24시간 실행

↓

야간 사용 안 함

↓

전기 낭비

↓

비용 증가
```

개발 서버는

평일 근무시간만 실행해도 충분합니다.

---

## 2.2.3 비용 절감 방법

### ① Right Sizing

적절한 크기의 EC2를 선택합니다.

예를 들어

CPU 사용률이

5%

밖에 되지 않는다면

```text
m7i.large

↓

t4g.small
```

처럼 작은 인스턴스로 변경할 수 있습니다.

---

### ② 사용하지 않는 리소스 제거

많은 기업이

다음과 같은 리소스를 삭제하지 않습니다.

* 미사용 EC2
* 사용하지 않는 EBS
* 오래된 Snapshot
* Elastic IP

이런 리소스도 비용이 발생합니다.

---

### ③ Savings Plans

장기간 사용할 경우

최대 72%

까지 할인받을 수 있습니다.

---

### ④ Reserved Instance

1년 또는 3년 예약하면

온디맨드보다 저렴합니다.

---

### ⑤ Spot Instance

남는 AWS 자원을 매우 저렴하게 사용할 수 있습니다.

최대

90%

까지 할인됩니다.

단,

언제든 회수될 수 있습니다.

AI 학습이나 Batch 작업에 적합합니다.

# 2.3 Sustainability (지속 가능성)

## 2.3.1 Sustainability란?

Sustainability는

**환경을 고려하면서 클라우드를 운영하는 것**입니다.

AWS에서는

친환경 데이터센터와

효율적인 자원 사용을 통해

탄소 배출을 줄이는 것을 목표로 합니다.

---

## 2.3.2 왜 중요한가?

예전에는

IT에서는 성능만 중요했습니다.

하지만

최근에는

ESG(Environment, Social, Governance)

경영이 중요해졌습니다.

기업들은

탄소 배출량도 함께 관리해야 합니다.

---

## 2.3.3 Sustainability의 핵심 원칙

### ① 필요한 만큼만 사용

불필요한 서버는 종료합니다.

---

### ② 서버리스 적극 활용

Lambda는

요청이 있을 때만 실행됩니다.

```text
요청 없음

↓

서버 없음

↓

전력 소비 없음

↓

요청 발생

↓

Lambda 실행

↓

작업 완료

↓

자동 종료
```

---

### ③ 최신 하드웨어 사용

AWS Graviton Processor는

전력 효율이 높습니다.

동일한 작업을

더 적은 전력으로 수행할 수 있습니다.

---

### ④ 리소스 최적화

CPU 사용률이

10%

인 서버는

더 작은 인스턴스로 변경합니다.

이는

비용 절감뿐 아니라

전력 절감 효과도 있습니다.

# 2.4 세 가지 핵심(Pillar) 비교

| 구분     | Performance Efficiency                | Cost Optimization                         | Sustainability                           |
| ------ | ------------------------------------- | ----------------------------------------- | ---------------------------------------- |
| 목적     | 필요한 성능을 효율적으로 제공                      | 최소 비용으로 운영                                | 환경을 고려한 운영                               |
| 핵심 질문  | 필요한 성능을 제공하는가?                        | 비용을 낭비하지 않는가?                             | 자원을 친환경적으로 사용하는가?                        |
| 대표 서비스 | Auto Scaling, CloudFront, ECS, Lambda | Cost Explorer, Budgets, Compute Optimizer | Lambda, Graviton, S3 Intelligent-Tiering |
| 주요 키워드 | 성능, 확장성, 최신 기술                        | 비용 절감, 사용량 분석, 예약 할인                      | 전력 효율, 탄소 배출 감소, 자원 최적화                  |

# 2.5 세 Pillar의 관계

학생들이 가장 많이 헷갈리는 부분이 바로 Performance Efficiency와 Cost Optimization입니다.

예를 들어 EC2 인스턴스를 Auto Scaling으로 운영하면,

* **Performance Efficiency** 측면에서는 사용자가 많을 때 자동으로 서버를 늘려 성능을 유지합니다.
* **Cost Optimization** 측면에서는 사용자가 줄면 서버를 자동으로 줄여 불필요한 비용을 절감합니다.
* **Sustainability** 측면에서는 사용하지 않는 서버를 종료하여 전력 소비와 탄소 배출을 줄입니다.

즉, **하나의 설계가 여러 Pillar에 동시에 기여할 수 있습니다.**

# 2.6 AWS CCP 시험 핵심 포인트

AWS CCP 시험에서는 다음과 같은 시나리오 문제가 자주 출제됩니다.

| 상황                                                        | 해당 Pillar                                        |
| ----------------------------------------------------------- | ------------------------------------------------ |
| CloudFront를 사용하여 전 세계 사용자에게 빠르게 콘텐츠 제공 | Performance Efficiency                           |
| Auto Scaling으로 트래픽 증가 시 EC2 자동 추가               | Performance Efficiency                           |
| 사용하지 않는 EBS 볼륨 삭제                                 | Cost Optimization                                |
| AWS Budgets로 예산 초과 알림 설정                           | Cost Optimization                                |
| Lambda를 사용하여 요청이 있을 때만 코드 실행                | Sustainability (동시에 Performance Efficiency에도 기여) |
| Compute Optimizer로 EC2 크기 추천                           | Cost Optimization 및 Sustainability               |

---

# 핵심 요약

* **Performance Efficiency**는 **필요한 성능을 가장 효율적인 자원으로 제공**하는 것이 목표입니다. Auto Scaling, CloudFront, Lambda, ECS/EKS 등이 대표적인 서비스입니다.
* **Cost Optimization**은 **불필요한 비용을 줄이고 필요한 곳에만 비용을 사용하는 것**이 핵심입니다. Right Sizing, Savings Plans, Reserved Instances, Cost Explorer 등이 중요한 요소입니다.
* **Sustainability**는 **환경과 에너지 효율을 고려한 클라우드 운영**을 의미하며, 서버리스 아키텍처, Graviton 프로세서, 리소스 최적화 등을 통해 실현할 수 있습니다.

# 2.1 왜 6개의 Pillar를 함께 고려해야 할까?

학생들이 가장 많이 하는 질문 중 하나는 다음과 같습니다.

> "6개의 Pillar 중에서 가장 중요한 것은 무엇인가요?"

정답은 **모두 중요하다**입니다.

예를 들어 보겠습니다.

자동차를 설계한다고 생각해 봅시다.

자동차에는 다음 요소가 필요합니다.

* 엔진(성능)
* 브레이크(안전)
* 연비(비용)
* 내구성(신뢰성)
* 정비 편의성(운영)
* 친환경성(지속 가능성)

브레이크만 좋은 자동차도 문제가 있고, 연비만 좋은 자동차도 좋은 자동차가 아닙니다.

클라우드도 마찬가지입니다.

AWS는 다음 여섯 가지를 균형 있게 고려하는 것을 권장합니다.

```text
       좋은 클라우드 시스템

                 ▲
                 │
      ┌────────────────────┐
      │ Operational        │
      │ Security           │
      │ Reliability        │
      │ Performance        │
      │ Cost               │
      │ Sustainability     │
      └────────────────────┘
```

한 가지 요소만 뛰어나서는 좋은 아키텍처라고 할 수 없습니다.

예를 들어 **Auto Scaling**을 보면 "EC2를 자동으로 늘려준다"에서 끝나는 것이 아니라,

* **Sustainability**: 불필요한 인스턴스를 종료하여 에너지 소비를 줄인다.

처럼 **여러 Pillar와 연결해서 생각하는 습관**을 가지면 시나리오형 문제를 훨씬 쉽게 해결할 수 있습니다.

# 3.1 Migration(마이그레이션)이란?

## Migration의 정의

**Migration(마이그레이션)** 이란 기존에 운영하던 IT 시스템을 새로운 환경으로 이전하는 과정을 의미합니다.

AWS에서는 일반적으로 **온프레미스(On-Premises)** 환경에서 AWS 클라우드로 시스템을 이전하는 것을 의미합니다.

즉,

> **"현재 운영 중인 애플리케이션, 데이터베이스, 서버 등을 AWS 클라우드 환경으로 이전하는 과정"**

입니다.

---

## On-Premises란?

On-Premises는 기업이 직접 서버를 구매하여 운영하는 방식입니다.

```text
            회사 전산실

      ┌────────────────────┐
      │     Web Server      │
      │     DB Server       │
      │     Storage         │
      │     Firewall        │
      └────────────────────┘

      모든 장비를 회사가 직접 관리
```

기업은 다음과 같은 업무를 직접 수행해야 합니다.

* 서버 구매
* 서버 설치
* 네트워크 구성
* 전원 관리
* 냉각 시스템 운영
* 장애 대응
* 보안 관리
* 백업 관리

---

## AWS Cloud 환경

```text
             AWS Cloud

          Amazon EC2

          Amazon RDS

          Amazon S3

          Amazon VPC

      AWS가 인프라를 관리
```

기업은 애플리케이션과 데이터에 집중하고, AWS는 인프라 운영을 담당합니다.

---

# 3.2 왜 기업은 AWS로 Migration하는가?

많은 기업이 클라우드로 이전하는 이유는 단순히 "최신 기술"이기 때문이 아닙니다.

비즈니스 측면에서 다음과 같은 장점이 있기 때문입니다.

---

## ① 초기 투자 비용(CAPEX) 감소

기존 방식

```text
서비스 시작

↓

서버 구매

↓

스토리지 구매

↓

네트워크 장비 구매

↓

수천만 원 투자
```

AWS 방식

```text
서비스 시작

↓

EC2 생성

↓

사용한 만큼만 과금
```

### 장점

* 초기 투자 비용 절감
* 빠른 서비스 시작
* 유휴 장비 없음

---

## ② 빠른 서비스 구축

기존 환경에서는 서버 구매부터 설치까지 몇 주가 걸릴 수 있습니다.

AWS에서는 EC2 인스턴스를 몇 분 안에 생성할 수 있습니다.

```text
On-Premises

구매 → 배송 → 설치 → 설정

(수주 소요)

↓

AWS

EC2 생성

(수분 소요)
```

---

## ③ 확장성(Scalability)

서비스 이용자가 증가하면 쉽게 리소스를 추가할 수 있습니다.

```text
사용자 증가

↓

Auto Scaling

↓

EC2 자동 추가
```

---

## ④ 글로벌 서비스

AWS는 전 세계 여러 리전에 서비스를 제공합니다.

한국에서 성공한 서비스를 미국이나 유럽으로 확장할 때도 동일한 방식으로 구축할 수 있습니다.

---

## ⑤ 운영 부담 감소

기업은 애플리케이션 개발에 집중하고,

AWS는 하드웨어, 전원, 네트워크 등의 인프라를 관리합니다.

---

# 3.3 모든 시스템을 동일한 방식으로 이전할 수 있을까?

아닙니다.

예를 들어 다음과 같은 시스템이 있다고 가정해 보겠습니다.

| 시스템       | 특징          |
| --------- | ----------- |
| 오래된 ERP   | 수정이 어려움     |
| 웹 쇼핑몰     | 최신 Java 기반  |
| Oracle DB | 라이선스 비용 높음  |
| 파일 서버     | 단순 저장소      |
| 메일 서버     | SaaS로 대체 가능 |

이 모든 시스템을 같은 방법으로 이전하는 것은 비효율적입니다.

그래서 AWS는 **Migration 6R 전략**을 제공합니다.

---

# 3.4 AWS Migration 6R Framework

Migration 6R은 애플리케이션 특성에 따라 가장 적합한 이전 전략을 선택하는 방법입니다.

```text
                Migration 6R

                     │
 ┌──────────┬──────────┬──────────┐
 │          │          │
Rehost  Replatform  Repurchase

 │          │          │

Refactor   Retain    Retire
```

6R은 다음과 같이 구성됩니다.

| 전략         | 의미                   |
| ---------- | -------------------- |
| Rehost     | 그대로 이전(Lift & Shift) |
| Replatform | 일부 수정하여 이전           |
| Repurchase | SaaS로 교체             |
| Refactor   | 애플리케이션 재설계           |
| Retain     | 현재 환경 유지             |
| Retire     | 서비스 종료               |

이번 장에서는 가장 많이 사용되는 **Rehost**와 **Replatform**을 학습합니다.

---

# 3.5 Rehost (Lift and Shift)

## Rehost란?

Rehost는 **애플리케이션을 거의 수정하지 않고 AWS로 그대로 이전하는 전략**입니다.

별칭으로 **Lift and Shift**라고도 합니다.

쉽게 말하면,

> **"들어서(Lift) 그대로 옮긴다(Shift)."**

라는 의미입니다.

---

## Rehost 구조

```text
On-Premises

VMware

↓

Windows Server

↓

Java Application

↓

AWS

EC2

↓

Windows Server

↓

Java Application
```

애플리케이션은 거의 변경하지 않습니다.

---

## Rehost의 특징

* 가장 빠른 이전 방법
* 위험이 적음
* 코드 수정이 거의 없음
* 기존 운영 방식 유지

---

## Rehost의 장점

### 빠른 Migration

수개월이 걸리는 프로젝트도 비교적 짧은 기간 내에 이전할 수 있습니다.

---

### 낮은 위험

애플리케이션을 수정하지 않기 때문에 예상치 못한 오류가 적습니다.

---

### 운영 방식 유지

기존 운영 경험을 그대로 활용할 수 있습니다.

---

## Rehost의 단점

AWS의 장점을 충분히 활용하지 못합니다.

예를 들어

기존에 직접 설치한 MySQL을 그대로 EC2에서 운영한다면

Amazon RDS의 자동 백업, 자동 패치, Multi-AZ 기능을 사용할 수 없습니다.

---

## 실제 사례

A사는 100대 이상의 Windows 서버를 운영하고 있었습니다.

서비스 중단을 최소화하기 위해

기존 VMware 서버를 EC2로 그대로 이전했습니다.

이를 통해 단기간에 데이터센터를 종료하고 AWS 환경으로 전환할 수 있었습니다.

---

## AWS 서비스

Rehost에 자주 사용하는 서비스

* **AWS Application Migration Service (MGN)**: 물리 서버나 가상 서버를 EC2로 복제하여 이전
* **Amazon EC2**: 기존 애플리케이션 실행 환경 제공
* **AWS Migration Hub**: 여러 마이그레이션 작업을 중앙에서 관리

---

# 3.6 Replatform (Lift, Tinker and Shift)

## Replatform이란?

Replatform은 **애플리케이션은 유지하면서 일부 구성 요소를 AWS 관리형 서비스로 변경하는 전략**입니다.

즉,

> **"조금만 수정하여 AWS의 장점을 활용한다."**

라는 의미입니다.

---

## Replatform 구조

```text
기존 환경

Application

↓

MySQL

↓

AWS

Application

↓

Amazon RDS
```

애플리케이션은 그대로 유지하지만 데이터베이스를 Amazon RDS로 이전하는 것이 대표적인 예입니다.

---

## Replatform의 특징

* 애플리케이션은 거의 수정하지 않음
* 일부 AWS 서비스 활용
* 운영 부담 감소
* 성능과 관리 효율 향상

---

## Replatform의 장점

### 관리 효율 향상

Amazon RDS를 사용하면 다음 기능을 자동으로 제공합니다.

* 자동 백업
* 자동 패치
* Multi-AZ
* 모니터링

---

### 운영 비용 절감

DB 서버를 직접 관리하지 않아도 되므로 운영 인력이 줄어듭니다.

---

### AWS 기능 활용

CloudWatch, IAM, KMS 등과 쉽게 연동할 수 있습니다.

---

## Replatform의 단점

* 일부 애플리케이션 수정 필요
* 테스트가 필요
* Rehost보다 시간이 더 소요될 수 있음

---

## 실제 사례

B사는 쇼핑몰 시스템을 AWS로 이전하면서

웹 서버는 EC2를 그대로 사용하고,

기존 MySQL 데이터베이스는 Amazon RDS MySQL로 이전했습니다.

이를 통해 데이터베이스 운영 부담을 크게 줄이고, 자동 백업과 Multi-AZ를 활용하여 안정성을 높였습니다.

---

## AWS 서비스

Replatform에서 자주 사용하는 서비스

* **Amazon RDS**: 관리형 관계형 데이터베이스
* **Amazon Aurora**: 고성능 관리형 데이터베이스
* **Amazon ElastiCache**: 관리형 캐시 서비스
* **AWS Database Migration Service (DMS)**: 데이터베이스 이전
* **AWS SCT (Schema Conversion Tool)**: 데이터베이스 스키마 변환

---

# 3.7 Rehost와 Replatform 비교

| 항목         | Rehost       | Replatform         |
| ---------- | ------------ | ------------------ |
| 애플리케이션 수정  | 거의 없음        | 일부 수정              |
| 이전 속도      | 매우 빠름        | 빠름                 |
| 위험도        | 낮음           | 중간                 |
| AWS 서비스 활용 | 낮음           | 중간                 |
| 운영 효율      | 기존과 유사       | 향상                 |
| 비용 절감 효과   | 제한적          | 비교적 높음             |
| 대표 예시      | VMware → EC2 | MySQL → Amazon RDS |

# 3.8 어떤 전략을 선택해야 할까?

다음과 같은 상황을 가정해 보겠습니다.

### 상황 1

데이터센터 계약이 3개월 후 종료됩니다.

서비스 중단 없이 빠르게 이전해야 합니다.

**추천 전략**

→ **Rehost**

이유: 애플리케이션 변경 없이 빠르게 이전할 수 있습니다.

---

### 상황 2

애플리케이션은 그대로 유지하고 싶지만 데이터베이스 운영 부담을 줄이고 싶습니다.

**추천 전략**

→ **Replatform**

이유: Amazon RDS와 같은 관리형 서비스를 활용하여 운영 효율을 높일 수 있습니다.

---

# 3.9 AWS CCP 시험 핵심 포인트

시험에서는 **Rehost와 Replatform의 차이**를 묻는 문제가 자주 출제됩니다.

| 상황                     | 적합한 전략     |
| ---------------------- | ---------- |
| 애플리케이션 수정 없이 빠르게 이전    | Rehost     |
| 데이터베이스를 Amazon RDS로 변경 | Replatform |
| EC2로 기존 서버를 그대로 이전     | Rehost     |
| 일부 AWS 관리형 서비스를 활용     | Replatform |

---

# 핵심 요약

* **Migration**은 기존 시스템을 AWS 클라우드로 이전하는 과정입니다.
* **Migration 6R**은 애플리케이션 특성에 맞는 최적의 이전 전략을 선택하기 위한 프레임워크입니다.
* **Rehost(Lift and Shift)**는 애플리케이션을 거의 수정하지 않고 빠르게 이전하는 전략으로, 일정이 촉박하거나 위험을 최소화해야 할 때 적합합니다.
* **Replatform(Lift, Tinker and Shift)**은 애플리케이션은 유지하면서 Amazon RDS와 같은 AWS 관리형 서비스를 도입하여 운영 효율과 안정성을 높이는 전략입니다.

| Rehost     | **그대로 이전(Lift & Shift)** | Amazon EC2, AWS Application Migration Service(MGN), AWS Migration Hub |
| Replatform | **조금 수정하여 AWS 기능 활용**    | Amazon RDS, Amazon Aurora, AWS DMS, AWS SCT                           |

>
> 시험에서는 **"코드를 거의 수정하지 않는다"**라는 문구가 나오면 **Rehost**를 먼저 떠올리고, **"관리형 서비스(RDS, Aurora 등)로 일부 구성 요소를 변경한다"**는 문구가 나오면 **Replatform**을 떠올리면 대부분의 문제를 쉽게 해결할 수 있습니다.

# 3.2 Repurchase (SaaS로 교체)

## Repurchase란?

Repurchase는 **기존 애플리케이션을 유지하지 않고 새로운 SaaS(Software as a Service) 제품으로 교체하는 전략**입니다.

쉽게 말하면,

> **"직접 운영하지 말고 이미 만들어진 클라우드 서비스를 사용하자."**

라는 전략입니다.

---

## 기존 방식

```text
회사 메일 서버

↓

Exchange Server

↓

직접 운영
```

---

## Repurchase 방식

```text
회사 메일 서버

↓

Microsoft 365

또는

Google Workspace
```

더 이상 메일 서버를 직접 운영하지 않습니다.

---

## 또 다른 예

```text
기존 CRM

↓

Salesforce
```

---

## 장점

### 운영 부담 제거

더 이상 서버를 관리하지 않아도 됩니다.

---

### 최신 기능 사용

SaaS 제공업체가 지속적으로 업데이트합니다.

---

### 빠른 도입

설치 과정이 거의 필요 없습니다.

---

## 단점

* 기존 기능과 다를 수 있음
* 사용자 교육 필요
* 구독 비용 발생
* 특정 SaaS 제품에 종속(Vendor Lock-in)될 수 있음

---

## 실제 사례

한 중소기업은 자체 메일 서버를 운영하고 있었습니다.

메일 서버 장애와 스팸 관리에 많은 시간이 소요되었습니다.

Microsoft 365로 이전한 후

* 메일 서버 제거
* 유지보수 감소
* 보안 향상

효과를 얻었습니다.

---

# 3.3 Refactor (Re-Architect)

## Refactor란?

Refactor는 **애플리케이션을 클라우드 환경에 맞게 새롭게 설계하는 전략**입니다.

Migration 6R 중에서 가장 많은 시간과 비용이 필요한 전략입니다.

하지만 AWS의 장점을 가장 많이 활용할 수 있습니다.

---

## 기존 구조

```text
Monolithic

Web

↓

Application

↓

Database
```

모든 기능이 하나의 애플리케이션에 포함되어 있습니다.

---

## Refactor 후

```text
API Gateway

↓

Lambda

↓

Microservice

↓

DynamoDB

↓

S3
```

클라우드 네이티브(Cloud-Native) 구조로 변경합니다.

---

## Refactor에서 많이 사용하는 기술

* Microservices
* Serverless
* Containers
* Event Driven Architecture

---

## 장점

### 최고의 확장성

서비스별로 독립적으로 확장할 수 있습니다.

---

### 운영 효율 향상

각 서비스가 독립적으로 배포됩니다.

---

### AWS 기능 적극 활용

* Lambda
* DynamoDB
* EventBridge
* SQS
* SNS
* ECS
* EKS

등 다양한 관리형 서비스를 활용할 수 있습니다.

---

## 단점

* 개발 비용 증가
* 개발 기간 증가
* 테스트 증가
* 기술 난이도 상승

---

## 실제 사례

전자상거래 기업이 기존 Java 기반 모놀리식 애플리케이션을 운영하고 있었습니다.

주문 서비스만 트래픽이 급증하는 문제가 발생했습니다.

Refactor를 통해

* 주문
* 회원
* 결제
* 상품

서비스를 각각의 마이크로서비스로 분리했습니다.

이후 주문 서비스만 독립적으로 확장할 수 있게 되어 성능과 운영 효율이 크게 향상되었습니다.

---

# 3.4 Retain

## Retain이란?

Retain은 **현재 시스템을 그대로 유지하는 전략**입니다.

즉,

> **"지금은 이전하지 않는다."**

라는 의미입니다.

---

## 왜 이전하지 않을까?

다음과 같은 이유가 있을 수 있습니다.

* 아직 사용 중인 레거시 시스템
* 라이선스 문제
* 규제 문제
* 기술적 제약
* 이전 비용이 너무 큼

---

## 사례

```text
은행 메인프레임

↓

현재 유지
```

금융권에서는 핵심 업무 시스템을 즉시 클라우드로 이전하지 않는 경우가 많습니다.

---

## 장점

* 위험이 없음
* 기존 시스템 유지
* 운영 방식 변화 없음

---

## 단점

* 클라우드 장점 활용 불가
* 유지보수 비용 지속
* 기술 부채 증가 가능

---

# 3.5 Retire

## Retire란?

Retire는 **더 이상 필요하지 않은 시스템을 폐기하는 전략**입니다.

쉽게 말하면,

> **"사용하지 않는 시스템은 과감하게 종료하자."**

입니다.

---

## 사례

회사에서 운영 중인 오래된 시스템

```text
사내 게시판

↓

10년 동안 사용 안 함

↓

Retire

↓

삭제
```

---

## 장점

* 운영 비용 절감
* 보안 위험 감소
* 관리 대상 감소

---

## 단점

삭제 전에

다음 사항을 반드시 확인해야 합니다.

* 데이터 보존 필요 여부
* 법적 보관 의무
* 다른 시스템과의 연계 여부

---

# 3.6 Migration 6R 전체 비교

| 전략         | 수정 수준 | 이전 속도  | 비용    | AWS 활용도 | 대표 사례                    |
| ---------- | ----- | ------ | ----- | ------- | ------------------------ |
| Rehost     | 매우 낮음 | 매우 빠름  | 낮음    | 낮음      | VMware → EC2             |
| Replatform | 낮음    | 빠름     | 중간    | 중간      | MySQL → Amazon RDS       |
| Repurchase | 제품 교체 | 중간     | 중간    | 높음      | Exchange → Microsoft 365 |
| Refactor   | 매우 높음 | 느림     | 높음    | 매우 높음   | Monolith → Microservices |
| Retain     | 없음    | 이전 안 함 | 유지 비용 | 없음      | 메인프레임 유지                 |
| Retire     | 없음    | 즉시 가능  | 매우 낮음 | 해당 없음   | 미사용 시스템 삭제               |

---

# 3.7 어떤 전략을 선택해야 할까?

다음은 실제 프로젝트에서 자주 발생하는 상황입니다.

| 상황                        | 추천 전략      | 이유           |
| ------------------------- | ---------- | ------------ |
| 데이터센터 계약 종료가 임박함          | Rehost     | 가장 빠르게 이전 가능 |
| 애플리케이션은 유지하고 DB만 관리형으로 변경 | Replatform | 운영 부담 감소     |
| 자체 메일 서버를 없애고 싶음          | Repurchase | SaaS 사용      |
| 새로운 서비스 개발                | Refactor   | 클라우드 네이티브 설계 |
| 레거시 시스템은 당분간 유지           | Retain     | 위험 최소화       |
| 더 이상 사용하지 않는 시스템          | Retire     | 비용 절감        |

---

# 3.8 AWS Migration 관련 주요 서비스

AWS는 마이그레이션을 지원하기 위한 다양한 서비스를 제공합니다.

## AWS Migration Hub

여러 마이그레이션 프로젝트를 한 곳에서 관리할 수 있는 서비스입니다.

```text
Migration Hub

↓

프로젝트 현황

↓

진행률 확인

↓

리소스 관리
```

---

## AWS Application Migration Service (MGN)

기존 물리 서버나 가상 서버를 AWS EC2로 이전하는 서비스입니다.

주로 **Rehost 전략**에서 사용됩니다.

특징

* 최소 다운타임
* 지속적 복제
* 자동 전환

---

## AWS Database Migration Service (DMS)

데이터베이스를 AWS로 이전하는 서비스입니다.

```text
Oracle

↓

AWS DMS

↓

Amazon RDS
```

특징

* 다운타임 최소화
* 다양한 DB 지원
* 지속적인 데이터 복제

---

## AWS Schema Conversion Tool (SCT)

데이터베이스 스키마를 다른 DB 엔진에 맞게 변환하는 도구입니다.

예를 들어 Oracle을 PostgreSQL이나 Amazon Aurora PostgreSQL로 이전할 때 사용합니다.

---

# 3.9 실제 기업 사례

A기업은 다음과 같은 시스템을 운영하고 있었습니다.

| 시스템       | 선택한 전략     |
| --------- | ---------- |
| ERP       | Rehost     |
| CRM       | Repurchase |
| 쇼핑몰       | Refactor   |
| Oracle DB | Replatform |
| 메인프레임     | Retain     |
| 오래된 그룹웨어  | Retire     |

모든 시스템을 하나의 전략으로 이전하지 않고, 업무 특성에 맞게 다양한 전략을 조합하여 사용했습니다.

# 3.10 AWS CCP 예상 문제

기존 애플리케이션을 수정하지 않고 EC2로 이전하려고 한다.

가장 적절한 전략은?

① Refactor

② Replatform

③ Rehost

④ Repurchase

**정답:** ③ Rehost

---

### 문제 2

# 3.11 AWS CCP 암기법

| 키워드     | 전략         |
| ------- | ---------- |
| 그대로 이전  | Rehost     |
| 조금 수정   | Replatform |
| SaaS 구매 | Repurchase |
| 완전히 재개발 | Refactor   |
| 그대로 유지  | Retain     |
| 폐기      | Retire     |

### 외우기 쉬운 순서

**"옮기고(Host) → 조금 고치고(Platform) → 사서 쓰고(Purchase) → 새로 만들고(Factor) → 유지하고(Tain) → 없앤다(Tire)"**

이 순서대로 기억하면 6R을 쉽게 암기할 수 있습니다.

# 제3장 핵심 정리

Migration 6R은 **기존 시스템을 AWS 클라우드로 이전할 때 어떤 전략을 사용할 것인지 결정하는 프레임워크**입니다.

중요한 점은 **모든 시스템에 하나의 전략을 적용하는 것이 아니라, 시스템의 특성에 따라 전략을 선택하거나 조합한다는 것**입니다.

예를 들어, ERP는 Rehost, 데이터베이스는 Replatform, 메일은 Repurchase, 신규 서비스는 Refactor, 규제가 있는 시스템은 Retain, 사용하지 않는 시스템은 Retire를 적용할 수 있습니다.

---

# AWS CCP 시험 합격 핵심 비교표

| 전략         | 핵심 키워드               | 수정 수준  | AWS 서비스 예시                                  |
| ---------- | -------------------- | ------ | ------------------------------------------- |
| Rehost     | 그대로 이전(Lift & Shift) | 거의 없음  | EC2, AWS Application Migration Service(MGN) |
| Replatform | 일부 최적화               | 일부 수정  | Amazon RDS, AWS DMS                         |
| Repurchase | SaaS로 교체             | 제품 교체  | Microsoft 365, Salesforce                   |
| Refactor   | 클라우드 네이티브 재설계        | 대규모 수정 | Lambda, ECS, EKS, DynamoDB                  |
| Retain     | 현재 유지                | 없음     | 기존 시스템 유지                                   |
| Retire     | 시스템 폐기               | 없음     | 미사용 시스템 제거                                  |

---

## 강사의 한마디

AWS CCP 시험에서는 **"얼마나 수정하는가?"**를 기준으로 생각하면 정답을 쉽게 찾을 수 있습니다.

* **수정이 거의 없다** → **Rehost**
* **조금 수정하여 AWS 관리형 서비스를 사용한다** → **Replatform**
* **기존 제품 대신 SaaS를 도입한다** → **Repurchase**
* **애플리케이션을 새로 설계한다** → **Refactor**
* **당분간 이전하지 않는다** → **Retain**
* **더 이상 사용하지 않아 제거한다** → **Retire**

이 사고방식을 익히면 Migration 6R 관련 시나리오 문제를 빠르고 정확하게 해결할 수 있습니다.

### 제3장 전체 마무리

제3장을 통해 **Migration 6R**을 학습했다면, 이제 AWS CCP에서 중요한 두 축을 모두 이해한 것입니다.

* **제2장(Well-Architected Framework)**: **AWS에서 어떻게 잘 설계할 것인가?**
* **제3장(Migration 6R)**: **기존 시스템을 AWS로 어떻게 이전할 것인가?**

실제 프로젝트에서는 **Migration 6R로 적절한 이전 전략을 선택한 후, Well-Architected Framework의 6개 Pillar를 기준으로 아키텍처를 설계하고 지속적으로 개선**하는 것이 AWS가 권장하는 모범 사례입니다.

# 4.2 Migration과 Architecture의 차이

많은 학생들이 Migration과 Architecture를 같은 개념으로 생각합니다.

그러나 두 개념은 목적이 다릅니다.

## Migration

Migration은 **현재 시스템을 AWS로 이전하는 과정**입니다.

질문은 다음과 같습니다.

> "어떻게 AWS로 이전할 것인가?"

예를 들면,

* Rehost
* Replatform
* Refactor

등의 전략을 선택하는 것이 Migration입니다.

---

## Architecture

Architecture는 **AWS에서 시스템을 어떻게 설계할 것인가?**를 의미합니다.

질문은 다음과 같습니다.

> "AWS에서 어떤 구조로 운영할 것인가?"

예를 들면,

* Multi-AZ
* Auto Scaling
* CloudFront
* IAM
* Lambda

등을 어떻게 조합하여 설계할 것인지를 결정하는 과정입니다.

---

## 쉽게 비유하기

새 집으로 이사한다고 생각해 보겠습니다.

### Migration

현재 살고 있는 집에서

새로운 집으로 이사하는 과정입니다.

```text id="iu1i7e"
기존 집

↓

짐 포장

↓

이사

↓

새 집
```

---

### Architecture

새 집에서

가구를 어디에 놓고

침실을 어떻게 꾸미고

인터넷을 어떻게 설치할지 설계하는 것입니다.

```text id="9sxrgm"
새 집

↓

가구 배치

↓

인터넷 설치

↓

가전 배치
```

즉,

Migration은 **이사**

Architecture는 **집 설계**

라고 생각하면 이해하기 쉽습니다.

---

# 4.3 AWS 프로젝트는 어떤 순서로 진행될까?

실제 기업에서는 다음과 같은 절차로 AWS 프로젝트를 진행합니다.

```text id="txo1cn"
① 현재 시스템 분석

↓

② Migration 전략 결정

↓

③ AWS 설계

↓

④ 구축

↓

⑤ 테스트

↓

⑥ 운영 및 개선
```

각 단계를 자세히 살펴보겠습니다.

---

## ① 현재 시스템 분석 (Assessment)

먼저 기존 시스템을 분석합니다.

예를 들어

| 시스템 | 현재 환경    |
| --- | -------- |
| 쇼핑몰 | VMware   |
| DB  | Oracle   |
| 메일  | Exchange |
| ERP | Linux    |

또한 다음 항목도 확인합니다.

* CPU 사용률
* 메모리 사용량
* 운영체제
* 라이선스
* 네트워크 구성
* 사용자 수
* 보안 요구사항

이 과정을 **Migration Assessment**라고 합니다.

---

## ② Migration 전략 결정

시스템마다 적절한 Migration 전략을 선택합니다.

예를 들어

| 시스템       | 전략         |
| --------- | ---------- |
| 쇼핑몰       | Rehost     |
| Oracle    | Replatform |
| 메일        | Repurchase |
| 신규 AI 서비스 | Refactor   |
| 메인프레임     | Retain     |
| 오래된 시스템   | Retire     |

이 단계에서는 "모든 시스템을 AWS로 옮길 것인가?"가 아니라 **"각 시스템에 가장 적합한 전략은 무엇인가?"**를 결정합니다.

---

## ③ AWS 아키텍처 설계

Migration 전략이 결정되면 AWS 환경을 설계합니다.

예를 들어 쇼핑몰 시스템을 AWS로 이전한다고 가정해 보겠습니다.

```text id="t8l2or"
                 Internet
                      │
               Amazon Route 53
                      │
             AWS WAF + AWS Shield
                      │
          Application Load Balancer
                      │
        ┌─────────────┴─────────────┐
        │                           │
    EC2 (AZ-1)                  EC2 (AZ-2)
        │                           │
        └─────────────┬─────────────┘
                      │
              Amazon RDS Multi-AZ
                      │
             Amazon S3 (이미지 저장)
```

이제 각 구성 요소가 어떤 Well-Architected Pillar와 관련이 있는지 살펴보겠습니다.

# 4.5 Migration과 Well-Architected의 관계

다음 그림은 두 개념의 관계를 잘 보여줍니다.

```text id="6jv6bz"
            기존 시스템
                  │
                  ▼
        Migration Assessment
                  │
                  ▼
          Migration 6R 선택
                  │
                  ▼
             AWS로 이전
                  │
                  ▼
    Well-Architected Framework 적용
                  │
                  ▼
         운영 및 지속적인 개선
```

### 핵심 포인트

* **Migration 6R**은 **"AWS로 어떻게 이전할 것인가?"**를 결정합니다.
* **Well-Architected Framework**는 **"AWS에서 어떻게 잘 운영할 것인가?"**를 결정합니다.

즉, 두 개념은 경쟁 관계가 아니라 **순차적으로 적용되는 보완 관계**입니다.

2. **Migration 6R**을 사용하여 시스템별 이전 전략을 결정합니다.
3. AWS 환경으로 이전합니다.
4. **Well-Architected Framework의 6개 Pillar**를 기준으로 아키텍처를 설계합니다.
5. 운영하면서 지속적으로 개선합니다.

즉,

> **Migration은 "AWS로 어떻게 갈 것인가?"를 결정하는 과정이고, Well-Architected Framework는 "AWS에서 어떻게 잘 운영할 것인가?"를 결정하는 기준입니다.**

이 흐름을 이해하는 것이 AWS CCP뿐만 아니라 실제 AWS 프로젝트를 수행할 때도 가장 중요한 출발점입니다.

---

## AWS CCP 암기 포인트

| 질문                            | 정답                                                                     |
| ----------------------------- | ---------------------------------------------------------------------- |
| **어떻게 AWS로 이전할 것인가?**         | **Migration 6R**                                                       |
| **AWS에서 어떻게 잘 설계하고 운영할 것인가?** | **Well-Architected Framework**                                         |
| **Migration 후 가장 먼저 고려할 것은?** | **Well-Architected Framework의 6개 Pillar 적용**                           |
| **프로젝트 순서**                   | **Assessment → Migration 6R → AWS 구축 → Well-Architected 적용 → 운영 및 개선** |

### 다음 장 예고

**제4-2장**에서는 **금융, 전자상거래, 병원, 게임, AI 서비스** 등 다양한 산업 분야를 사례로 하여, **어떤 Migration 전략을 선택하고 어떤 Well-Architected Pillar를 적용하는지**를 실제 프로젝트 시나리오 중심으로 학습하겠습니다.

# 4.1 실무에서는 하나의 전략만 사용할까?

지금까지 Migration 6R을 각각 배웠습니다.

하지만 실제 기업에서는 **하나의 전략만 사용하는 경우는 거의 없습니다.**

예를 들어 한 기업에는 다음과 같은 시스템이 함께 존재합니다.

* 메일

* AI 서비스
* 파일 서버
* 그룹웨어

이 모든 시스템을 **Rehost**만 사용하는 것은 비효율적입니다.

실제 프로젝트에서는 다음과 같이 여러 전략을 함께 사용합니다.

           │             │            │

이것이 실무에서 말하는 **Hybrid Migration Strategy**입니다.

# 4.12 실무에서 기억해야 할 핵심 사항

실제 프로젝트에서는 **Migration 6R**과 **Well-Architected Framework**를 별개로 생각하지 않습니다.

* 먼저 **Migration 6R**을 통해 "어떻게 이전할 것인가?"를 결정합니다.
* 이후 **Well-Architected Framework**를 기준으로 "어떻게 설계하고 운영할 것인가?"를 고민합니다.
* 마지막으로 운영 과정에서 **Well-Architected Review**를 반복하여 지속적으로 개선합니다.

즉, **Migration은 시작점**, **Well-Architected Framework는 운영의 기준**이라고 이해하면 됩니다.

| Performance Efficiency | 성능 최적화 | 필요한 성능을 제공하는가?       | CloudFront, Lambda, ECS, EKS              | Scalability       |
| Cost Optimization      | 비용 절감  | 비용을 낭비하지 않는가?        | Cost Explorer, Budgets, Savings Plans     | Cost Saving       |
| Sustainability         | 친환경 운영 | 에너지 효율적으로 운영하는가?     | Graviton, Lambda, Compute Optimizer       | Energy Efficiency |

---

# 4.3 Migration 6R 최종 정리

| 전략         | 의미                   | 수정 수준  | AWS 활용도 | 대표 사례                    |
| ---------- | -------------------- | ------ | ------- | ------------------------ |
| Rehost     | 그대로 이전(Lift & Shift) | 거의 없음  | 낮음      | VMware → EC2             |
| Replatform | 일부 최적화               | 일부 수정  | 중간      | MySQL → Amazon RDS       |
| Repurchase | SaaS로 교체             | 제품 변경  | 높음      | Exchange → Microsoft 365 |
| Refactor   | 클라우드 네이티브 재설계        | 대규모 수정 | 매우 높음   | Monolith → Microservices |
| Retain     | 현재 유지                | 없음     | 없음      | 메인프레임 유지                 |
| Retire     | 시스템 제거               | 없음     | 해당 없음   | 사용하지 않는 시스템 삭제           |

AWS CCP에서는 서비스 이름만 암기하는 것이 아니라 **서비스가 어떤 설계 원칙(Pillar)에 가장 밀접하게 관련되는지** 이해하는 것이 중요합니다.

| CloudFormation        | Operational Excellence              | 인프라 자동화(IaC)    |

| CloudTrail            | Security                            | 감사 로그           |

| Elastic Load Balancer | Reliability                         | 부하 분산           |
| EC2 Auto Scaling      | Reliability / Performance           | 자동 확장 및 장애 대응   |
| Amazon RDS Multi-AZ   | Reliability                         | 데이터베이스 이중화      |
| Route 53              | Reliability                         | 장애 조치(Failover) |

# 4.5 AWS CCP에서 자주 헷갈리는 개념 비교

## Rehost vs Replatform

| Rehost | Replatform       |
| ------ | ---------------- |
| 그대로 이전 | 일부 수정            |
| EC2 사용 | RDS 등 관리형 서비스 활용 |
| 가장 빠름  | 운영 효율 향상         |

---

## Scalability vs Elasticity

| Scalability | Elasticity                |
| ----------- | ------------------------- |
| 시스템의 확장 능력  | 트래픽에 따라 자동으로 늘어나고 줄어드는 능력 |
| 미래 성장을 대비   | 현재 부하에 실시간 대응             |

예를 들어,

* EC2 인스턴스를 더 큰 타입으로 변경하는 것은 **Scalability**입니다.
* Auto Scaling으로 EC2 수를 자동 조절하는 것은 **Elasticity**입니다.

---

## High Availability vs Fault Tolerance

| High Availability | Fault Tolerance   |
| ----------------- | ----------------- |
| 장애 발생 시 빠르게 복구    | 장애가 발생해도 중단 없이 운영 |
| Multi-AZ          | 이중화, 다중 리전 등      |

---

## CAPEX vs OPEX

| CAPEX    | OPEX                     |
| -------- | ------------------------ |
| 초기 투자 비용 | 운영 비용                    |
| 서버 구매    | 사용량 기반 과금(Pay-as-you-go) |

AWS는 **CAPEX를 OPEX로 전환**하는 대표적인 클라우드 서비스입니다.

# 4.9 AWS CCP 합격 전략

AWS CCP 시험은 서비스 이름을 모두 암기하는 시험이 아닙니다.

다음 세 가지를 이해하는 것이 중요합니다.

### ① 문제에서 무엇을 묻는가?

예를 들어,

* 보안
* 성능
* 비용
* 운영
* 장애 대응

중 무엇이 핵심인지 먼저 파악합니다.

---

### ② 어떤 Well-Architected Pillar와 관련 있는가?

예를 들어,

* IAM → Security
* CloudFront → Performance Efficiency
* Multi-AZ → Reliability

처럼 연결해서 생각합니다.

---

### ③ 어떤 AWS 서비스가 적합한가?

예를 들어,

* 자동 배포 → CodePipeline
* 비용 분석 → Cost Explorer
* 데이터 암호화 → KMS
* 콘텐츠 캐싱 → CloudFront

처럼 서비스와 목적을 연결합니다.

# 강사의 마무리

AWS CCP는 AWS의 모든 서비스를 깊이 있게 다루는 시험이 아니라, **클라우드의 기본 개념과 AWS의 대표 서비스, 그리고 좋은 클라우드 아키텍처를 이해하고 있는지를 평가하는 시험**입니다.

특히 설계 영역에서는 다음의 두 가지를 항상 연결해서 생각하는 습관을 가지면 대부분의 시나리오 문제를 해결할 수 있습니다.

1. **Migration 6R**
   → *기존 시스템을 AWS로 어떻게 이전할 것인가?*

2. **Well-Architected Framework**
   → *AWS에서 어떻게 안정적이고 효율적으로 운영할 것인가?*

그리고 마지막으로 한 가지를 더 기억하세요.

> **AWS 서비스는 암기 대상이 아니라 "문제를 해결하기 위한 도구"입니다.**

문제에서 요구하는 것이 **보안인지, 성능인지, 비용 절감인지, 운영 자동화인지**를 먼저 파악하면, 어떤 AWS 서비스를 선택해야 하는지가 자연스럽게 보입니다.

현재까지 작성한 **1~4장**은 약 **4만 자** 수준의 교재입니다. 이 교재를 더 실무적으로 활용하려면 다음 내용을 추가하는 것을 추천합니다.

이렇게 구성하면 대학 강의나 기업 교육에서도 사용할 수 있는 **AWS CCP 완성형 교재**가 됩니다.
