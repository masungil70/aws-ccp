# 애플리케이션 실행과 서버리스(Serverless) 아키텍처 전체 구조

```
                사용자
                   │
             API Gateway
                   │
        ┌──────────┴──────────┐
        │                     │
      Lambda               ECS / EKS
                               │
                          Fargate 실행
                               │
                          Application
                               │
                     EventBridge 이벤트 발생
                               │
                      Step Functions 실행
                               │
                      여러 AWS 서비스 연동
```

즉,

* API Gateway → 요청을 받는다.
* ECS/EKS → 컨테이너를 실행한다.
* Fargate → 서버 없이 컨테이너를 실행한다.
* EventBridge → 이벤트를 전달한다.
* Step Functions → 여러 작업을 순서대로 실행한다.

---

# 1. ECS (Elastic Container Service)

## ECS란?

ECS는

> **AWS가 만든 컨테이너(Container) 관리 서비스**

입니다.

Docker Container를 쉽게 실행하고 관리하도록 만들어진 서비스입니다.

쉽게 말하면

> "Docker를 AWS에서 관리해주는 서비스"

입니다.

---

## 왜 필요한가?

Docker를 여러 대의 서버에서 실행한다고 생각해봅시다.

```
Docker Host 1
Docker Host 2
Docker Host 3
```

어느 서버에 실행할지 ?

몇 개 실행할지 ?

죽으면 다시 실행할지 ?

직접 관리해야 합니다.

ECS가 대신 관리합니다.

---

## ECS 구성

```
ECS Cluster
     │
     ├── Service
     │      │
     │      ├── Task
     │      ├── Task
     │      └── Task
```

### Cluster

컨테이너들이 실행되는 공간

### Task

실제 Docker Container

### Service

Task 개수를 유지

예)

```
Task 3개 유지
```

한 개 죽으면

자동으로 다시 생성

---

## 장점

* AWS와 완벽하게 통합
* 배우기 쉬움
* 관리가 쉬움
* 비용 저렴

---

## 언제 사용할까?

* Docker 기반 웹 서비스
* API 서버
* Batch 작업

---

# 2. EKS (Elastic Kubernetes Service)

## EKS란?

EKS는

> Kubernetes를 AWS가 관리해주는 서비스

입니다.

Kubernetes는

컨테이너 오케스트레이션의 표준입니다.

---

## Kubernetes란?

Docker가 하나의 컨테이너를 실행한다면

Kubernetes는

수천 개의 컨테이너를 관리합니다.

예)

```
100개의 컨테이너

자동 배포

자동 복구

자동 확장

자동 업데이트
```

---

## EKS 구조

```
EKS Cluster

     │

Node Group

     │

Pods

     │

Container
```

---

## 장점

* Kubernetes 표준
* 멀티 클라우드 가능
* 확장성 최고
* 기업에서 많이 사용

---

## 단점

조금 어렵습니다.

초보자에게는 ECS보다 난이도가 높습니다.

---

## 언제 사용할까?

* Kubernetes 환경
* 대규모 서비스
* 멀티 클라우드

---

# 3. Fargate

## Fargate란?

Fargate는

> 서버를 관리하지 않고 컨테이너를 실행하는 서비스

입니다.

보통은

```
EC2 생성

↓

OS 설치

↓

Docker 설치

↓

Container 실행
```

하지만

Fargate는

```
Container만 올리면 끝
```

---

## 관리 대상 비교

### EC2 기반

```
서버

OS

패치

Docker

모두 관리
```

### Fargate

```
Container만 관리
```

---

## 장점

* 서버 관리 없음
* 자동 확장
* 사용한 만큼 비용

---

## ECS와 함께 사용

```
ECS

↓

Launch Type

↓

EC2

또는

Fargate
```

즉

ECS는 관리 서비스

Fargate는 실행 환경입니다.

---

# ECS + Fargate

```
ECS

↓

Task

↓

Fargate 실행
```

---

# EKS + Fargate

```
Kubernetes Pod

↓

Fargate 실행
```

---

# 4. API Gateway

## API Gateway란?

API Gateway는

> API의 출입문

입니다.

모든 요청을 먼저 받습니다.

```
Client

↓

API Gateway

↓

Backend
```

---

## 역할

* 인증
* 권한
* 요청 제한
* Logging
* Routing

---

## 연결 가능한 서비스

```
API Gateway

↓

Lambda

↓

ECS

↓

EKS

↓

EC2
```

---

## 장점

* 서버리스
* 확장 자동
* HTTPS 제공

---

## 사용 사례

```
모바일 앱

↓

API Gateway

↓

Lambda
```

---

# 5. EventBridge

## EventBridge란?

EventBridge는

> 이벤트(Event)를 전달하는 서비스

입니다.

이벤트가 발생하면

자동으로 다른 서비스를 실행합니다.

---

## 예시

```
S3 업로드

↓

EventBridge

↓

Lambda 실행
```

---

또는

```
EC2 종료

↓

EventBridge

↓

SNS 알림
```

---

## 이벤트 기반(Event Driven)

```
무언가 발생

↓

자동 실행
```

---

## 장점

* 서버리스
* 다양한 AWS 서비스 연동
* 스케줄 실행 가능

---

# 6. Step Functions

## Step Functions란?

Step Functions는

> 여러 작업을 순서대로 실행하는 Workflow 서비스

입니다.

---

예)

회원가입

```
회원 생성

↓

이메일 발송

↓

포인트 지급

↓

SMS 발송
```

이 과정을 자동으로 실행합니다.

---

## 시각적 Workflow

```
Step1

↓

Step2

↓

Step3

↓

Step4
```

실패하면

```
재시도

또는

오류 처리
```

도 가능합니다.

---

## 장점

* Workflow 관리
* 재시도
* 오류 처리
* 시각적 관리

---

## 사용 사례

AI 서비스

```
파일 업로드

↓

OCR

↓

번역

↓

요약

↓

DB 저장
```

                              │

# 서비스 비교

| 서비스         | 역할              | 서버 관리               | 대표 용도             | CCP 중요도 |
| -------------- | ----------------- | ----------------------- | --------------------- | ------- |
| ECS            | AWS 컨테이너 관리     | 필요(EC2) 또는 불필요(Fargate) | Docker 애플리케이션 실행      | ★★★★★   |
| EKS            | Kubernetes 관리   | 필요(EC2) 또는 불필요(Fargate) | Kubernetes 기반 애플리케이션  | ★★★★☆   |
| Fargate        | 서버리스 컨테이너 실행 환경 | 불필요                     | ECS/EKS에서 컨테이너 실행     | ★★★★★   |
| API Gateway    | API 관리 및 진입점    | 불필요                     | REST API, HTTP API 제공 | ★★★★★   |
| EventBridge    | 이벤트 전달           | 불필요                     | 이벤트 기반 자동화            | ★★★★☆   |
| Step Functions | 워크플로우 오케스트레이션   | 불필요                     | 여러 작업의 순차 실행 및 오류 처리  | ★★★★☆   |

---

# ECS vs EKS

| 항목            | ECS              | EKS                              |
| --------------- | ---------------- | -------------------------------- |
| 기반 기술       | AWS 전용         | Kubernetes                       |
| 학습 난이도     | 쉬움             | 어려움                           |
| 관리 편의성     | 매우 쉬움        | 복잡함                           |
| 멀티 클라우드   | 불가             | 가능                             |
| AWS 서비스 연동 | 매우 우수        | 우수                             |
| 추천 대상       | 초보자, AWS 중심 | Kubernetes 경험자, 대규모 서비스 |

**시험 포인트:** ECS는 AWS 자체 컨테이너 오케스트레이션 서비스이고, EKS는 Kubernetes를 관리형으로 제공하는 서비스입니다.

---

# ECS vs Fargate

| 항목          | ECS                                 | Fargate            |
| ------------- | ----------------------------------- | ------------------ |
| 서비스 유형   | 컨테이너 관리 서비스                | 컨테이너 실행 환경 |
| 역할          | 작업(Task)과 서비스(Service)를 관리 | 컨테이너를 실행    |
| EC2 필요 여부 | EC2 또는 Fargate 선택 가능          | EC2 불필요         |
| 서버 관리     | 선택 사항                           | 없음               |

**시험 포인트:** Fargate는 ECS를 대체하는 서비스가 아니라, ECS 또는 EKS에서 사용할 수 있는 **서버리스 실행 환경**입니다.

---

# EventBridge vs Step Functions

| 항목                | EventBridge              | Step Functions                          |
| ------------------- | ------------------------ | --------------------------------------- |
| 목적                | 이벤트 전달              | 워크플로우 실행                         |
| 실행 방식           | 이벤트 발생 시 대상 호출 | 정의된 순서대로 단계 실행               |
| 순차 처리           | 불가능                   | 가능                                    |
| 조건 분기           | 제한적                   | 가능                                    |
| 재시도 및 오류 처리 | 기본 수준                | 강력한 지원                             |
| 대표 사례           | S3 업로드 → Lambda 실행  | 주문 처리, AI 파이프라인, 승인 프로세스 |

**시험 포인트:** EventBridge는 "이벤트를 연결하는 서비스", Step Functions는 "여러 작업을 순서대로 제어하는 서비스"로 구분하면 이해하기 쉽습니다.

---

# API Gateway vs EventBridge

| 항목        | API Gateway                         | EventBridge                                    |
| ----------- | ----------------------------------- | ---------------------------------------------- |
| 트리거      | 사용자 또는 애플리케이션의 API 호출 | AWS 서비스 또는 애플리케이션에서 발생한 이벤트 |
| 목적        | API 요청 처리                       | 이벤트 기반 자동화                             |
| 동기/비동기 | 주로 동기(Synchronous)              | 비동기(Asynchronous)                           |
| 대표 사례   | 모바일 앱 → API 호출                | S3 파일 업로드 → 후속 작업 자동 실행           |

---

# AWS CCP 시험 핵심 암기 포인트

| 키워드             | 반드시 기억할 내용                                                             |
| ------------------ | ------------------------------------------------------------------------------ |
| **ECS**            | AWS의 관리형 컨테이너 오케스트레이션 서비스(Docker 중심)                       |
| **EKS**            | AWS의 관리형 Kubernetes 서비스                                                 |
| **Fargate**        | EC2 서버를 직접 관리하지 않는 서버리스 컨테이너 실행 환경                      |
| **API Gateway**    | API의 진입점으로 인증, 라우팅, 요청 관리 기능 제공                             |
| **EventBridge**    | 이벤트 기반으로 AWS 서비스와 애플리케이션을 연결하는 서비스                    |
| **Step Functions** | 여러 작업을 순서대로 실행하고 오류 처리 및 재시도를 지원하는 워크플로우 서비스 |

## 한 줄로 정리

* **ECS**: Docker 컨테이너를 AWS에서 쉽게 운영한다.
* **EKS**: Kubernetes를 AWS에서 관리형으로 운영한다.
* **Fargate**: 서버를 신경 쓰지 않고 컨테이너만 실행한다.
* **API Gateway**: API 요청을 받아 백엔드 서비스로 전달하는 출입문이다.
* **EventBridge**: 이벤트가 발생하면 필요한 작업을 자동으로 시작한다.
* **Step Functions**: 여러 서비스를 하나의 비즈니스 프로세스로 연결하고 순차적으로 실행한다.

이 여섯 가지 서비스는 함께 사용되는 경우가 많습니다. 예를 들어 **API Gateway → ECS(Fargate) → EventBridge → Step Functions**와 같은 구성을 통해 API 요청 처리부터 이벤트 기반 자동화, 복잡한 업무 프로세스 실행까지 AWS에서 효율적으로 구현할 수 있습니다.
