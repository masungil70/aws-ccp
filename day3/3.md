# 데이터베이스(Database) 서비스 이해


많은 학생들이 **"RDS와 Aurora는 뭐가 다르지?"**, **"DynamoDB는 왜 SQL이 안되지?"**, **"Redshift는 데이터베이스인가?"**, **"ElastiCache는 왜 DB라고 부르지 않을까?"** 라는 질문을 많이 합니다.

이번 장에서는 각각의 서비스가 **무엇을 저장하는지**, **어떤 상황에서 사용하는지**, **서로 어떤 차이가 있는지** 중심으로 설명하겠습니다.

# 1. Amazon RDS (Relational Database Service)

## 개념

RDS는 AWS에서 제공하는 **관리형 관계형 데이터베이스 서비스(Managed Relational Database)** 입니다.

직접 DB 서버를 설치하지 않아도 됩니다.

AWS가

* 설치
* 패치
* 백업
* 장애복구
* 모니터링

등을 자동으로 수행합니다.

---

## 쉽게 이해하기

학교에서 학생 정보를 저장한다고 생각해 봅시다.

| 학번  | 이름   | 학과   |
| ----- | ------ | ------ |
| 1001  | 홍길동 | 컴퓨터 |
| 1002  | 김철수 | 전자   |

이처럼

* 행(Row)
* 열(Column)

형태로 저장되는 것이 관계형 데이터베이스입니다.

---

## 지원 DB 엔진

RDS에서는 다양한 데이터베이스를 선택할 수 있습니다.

* MySQL
* PostgreSQL
* MariaDB
* Oracle
* SQL Server

즉,

AWS가 대신 관리해주는 데이터베이스 서비스입니다.

---

## 특징

* SQL 사용
* 자동 백업
* Multi-AZ 지원
* Read Replica 지원
* 자동 패치
* 자동 장애 복구

---

## 사용하는 예

* 쇼핑몰
* 학교 시스템
* ERP
* 게시판
* 회원관리

---

# 2. Amazon Aurora

## 개념

Aurora는

**AWS가 직접 만든 차세대 관계형 데이터베이스**

입니다.

MySQL과 PostgreSQL을 그대로 사용할 수 있지만 성능은 훨씬 뛰어납니다.

---

## 특징

AWS 시험에서는 반드시 기억해야 합니다.

Aurora는

> "RDS의 한 종류"

입니다.

즉

```
RDS
 ├── MySQL
 ├── PostgreSQL
 ├── Oracle
 ├── SQL Server
 └── Aurora
```

Aurora도 RDS 서비스 안에서 생성합니다.

---

## 왜 빠를까?

일반 MySQL

```
DB
 │
 ▼
디스크
```

Aurora

```
DB
 │
 ▼
AWS 분산 스토리지
(6개 복제)

AZ1
AZ2
AZ3
```

데이터가

3개의 AZ에

6개의 복사본으로 저장됩니다.

따라서

* 매우 빠름
* 장애에 강함
* 자동 복구

가 가능합니다.

---

## 성능

AWS 공식 기준

* MySQL보다 최대 5배
* PostgreSQL보다 최대 3배

정도 빠른 성능을 제공합니다.

---

## 사용하는 예

* 대형 쇼핑몰
* 게임
* 금융
* SaaS 서비스

---

# RDS vs Aurora

| 항목          | RDS  | Aurora    |
| ------------- | ---- | --------- |
| 관리형 DB     | O    | O         |
| 관계형 DB     | O    | O         |
| SQL 사용      | O    | O         |
| AWS 자체 개발 | X    | O         |
| 속도          | 보통 | 매우 빠름 |
| 고가용성      | 좋음 | 매우 좋음 |
| 자동 복제     | 기본 | 6개 복제  |

---

# 3. Amazon DynamoDB

## 개념

DynamoDB는

AWS의 **NoSQL 데이터베이스**입니다.

SQL이 아니라

Key-Value 방식으로 저장합니다.

---

## 예

학생 정보

```
Key

1001

Value

{
 이름:"홍길동",
 학과:"컴퓨터",
 학년:3
}
```

---

## 특징

* 매우 빠름
* 서버 관리 없음
* 자동 확장
* SSD 사용
* Millisecond 응답속도

---

## 언제 사용할까?

* 게임
* 모바일 앱
* IoT
* 채팅
* 로그인 정보

---

## 장점

수백만 명이 동시에 접속해도 자동으로 확장됩니다.

---

# RDS와 DynamoDB 차이

| 항목           | RDS    | DynamoDB    |
| -------------- | ------ | ---------   |
| 데이터 형태    | Table  | Key-Value   |
| SQL            | O      | X           |
| 관계(Relation) | O      | X           |
| 자동 확장      | 제한적 | 매우 뛰어남 |
| 속도           | 빠름   | 매우 빠름   |

---

# 4. Amazon Redshift

## 개념

Redshift는

데이터를 저장하는 목적이 아닙니다.

정확히는

**데이터 분석(Data Warehouse)** 을 위한 서비스입니다.

---

## 예

회사에서

10년간의 판매 데이터를 분석한다고 가정해 봅시다.

```
2021
2022
2023
2024
2025

↓

매출 분석
지역 분석
상품 분석
```

이런 대량 분석이 Redshift입니다.

---

## 특징

* PB급 저장
* OLAP
* BI 연동
* SQL 사용
* Column 기반 저장

---

## 사용하는 예

* 매출 분석
* 고객 분석
* AI 데이터 분석
* 경영 보고서

---

# RDS와 Redshift 차이

| 항목        | RDS       | Redshift |
| --------    | -----     | -------- |
| 목적        | 업무 처리 | 분석     |
| OLTP        | O         | X        |
| OLAP        | X         | O        |
| 실시간 거래 | O         | X        |
| 대용량 분석 | △         | O        |


AWS CCP에서는 **OLTP와 OLAP의 차이**를 이해하는 것이 매우 중요합니다. 특히 **RDS/Aurora ↔ Redshift**를 구분하는 대표적인 개념이기 때문입니다.

---

OLTP (Online Transaction Processing)는 **실시간 거래(Transaction)를 처리하는 데이터베이스 시스템**입니다. 즉,**현재 발생하는 업무를 빠르고 정확하게 처리하는 것이 목적**입니다.

대표적으로

* 주문
* 결제
* 회원가입
* 로그인
* 송금

같이 매 순간 발생하는 데이터를 처리합니다.

OLAP (Online Analytical Processing)는 **데이터를 분석하기 위한 시스템**입니다. 즉, **이미 저장된 대량의 데이터를 분석하는 것이 목적**입니다.

---

# OLTP vs OLAP

| 비교 항목   | OLTP                   | OLAP             |
| ----------- | ---------------------- | ----------       |
| 목적        | 업무 처리              | 데이터 분석      |
| 데이터      | 현재 데이터            | 과거 데이터      |
| 작업        | INSERT, UPDATE, DELETE | SELECT           |
| 데이터 양   | 적음                   | 매우 많음        |
| 응답시간    | 매우 빠름(ms)          | 수 초~수 분 가능 |
| 사용자      | 고객, 직원             | 관리자, 분석가   |
| 대표 서비스 | RDS, Aurora            | Redshift         |

---

# 5. Amazon ElastiCache

## 개념

ElastiCache는 메모리(RAM)에 데이터를 저장하는 서비스입니다.

---

## 왜 사용할까?

웹사이트에서는

같은 데이터를 계속 조회합니다.

예)

학생 목록

```
DB 조회

↓

학생 1000명
```

매번 DB를 조회하면 느립니다.

따라서

메모리에 저장합니다.

```
Application

↓

ElastiCache

↓

RDS
```

---

## 특징

* 매우 빠름
* 메모리 저장
* 응답속도 감소
* DB 부하 감소

---

## 지원 엔진

* Redis
* Memcached

---

## 사용하는 예

* 로그인 세션
* 인기 게시글
* 쇼핑몰 상품
* 게임 랭킹

---

# ElastiCache 동작 과정

```
사용자
   ↓
Application
   ↓
ElastiCache
│
├── 있으면(Cache Hit)
│        ↓
│      바로 응답
│
└── 없으면(Cache Miss)
        ↓
      RDS 조회
        ↓
   Cache 저장
        ↓
   사용자 응답
```

---

# 서비스별 사용 목적

| 서비스      | 목적                    |
| ----------- | ----------------------- |
| RDS         | 일반 업무 데이터 저장   |
| Aurora      | 고성능 업무 데이터 저장 |
| DynamoDB    | 대규모 NoSQL 서비스     |
| Redshift    | 빅데이터 분석           |
| ElastiCache | 속도 향상을 위한 캐시   |

---

# 시험에서 자주 나오는 비교

| 질문                   | 정답                   |
| ---------------------- | ---------------------- |
| SQL 데이터베이스?      | RDS, Aurora, Redshift  |
| NoSQL?                 | DynamoDB               |
| 메모리 캐시?           | ElastiCache            |
| 데이터 분석?           | Redshift               |
| 웹 서비스 DB?          | RDS                    |
| 고성능 관계형 DB?      | Aurora                 |
| 자동 확장이 뛰어난 DB? | DynamoDB               |
| 로그인 세션 저장?      | ElastiCache            |
| 쇼핑몰 주문 데이터?    | RDS 또는 Aurora        |
| 실시간 랭킹?           | DynamoDB + ElastiCache |

---

# 서비스 간 관계

```
                  사용자
                     │
                 Web / Mobile
                     │
              Application
      ┌──────────────┼───────────────┐
      │              │               │
      ▼              ▼               ▼
 ElastiCache      RDS/Aurora     DynamoDB
 (캐시)          (관계형 DB)     (NoSQL)
                      │
                      ▼
             ETL / 데이터 적재
                      │
                      ▼
                 Redshift
             (데이터 분석용)
```

* **ElastiCache**는 RDS나 Aurora 앞단에서 자주 조회되는 데이터를 메모리에 저장하여 성능을 높입니다.
* **RDS/Aurora**는 주문, 회원, 결제와 같은 트랜잭션 데이터를 저장합니다.
* **DynamoDB**는 대규모 사용자 세션, 게임 상태, IoT 데이터 등 빠른 읽기/쓰기가 필요한 데이터를 저장합니다.
* 운영 데이터는 ETL 과정을 거쳐 **Redshift**에 적재되어 통계, BI, 데이터 분석에 활용됩니다.

# AWS CCP 시험 핵심 암기 포인트

| 서비스          | 핵심 키워드                              | 대표 사용 사례                  |
| --------------- | ---------------------------------------- | ----------------------------    |
| **RDS**         | 관리형 관계형 데이터베이스, SQL          | 웹사이트, ERP, 회원관리         |
| **Aurora**      | AWS 고성능 관계형 데이터베이스, RDS 기반 | 대규모 웹 서비스, 금융, SaaS    |
| **DynamoDB**    | NoSQL, Key-Value, 자동 확장              | 게임, 모바일, IoT               |
| **Redshift**    | 데이터 웨어하우스, OLAP, 분석            | BI, 매출 분석, 데이터 분석      |
| **ElastiCache** | 메모리 캐시, Redis, Memcached            | 세션 관리, 캐싱, 응답 속도 향상 |

## 한 줄 암기법

* **RDS** → **"일반적인 SQL 데이터베이스"**
* **Aurora** → **"더 빠르고 더 안정적인 RDS"**
* **DynamoDB** → **"NoSQL + 자동 확장"**
* **Redshift** → **"분석 전용 데이터 웨어하우스"**
* **ElastiCache** → **"데이터베이스가 아니라 메모리 캐시"**

이 다섯 가지 서비스의 **저장 방식(디스크 vs 메모리), 데이터 모델(SQL vs NoSQL), 사용 목적(운영 vs 분석 vs 캐시)**을 구분할 수 있으면 AWS CCP 시험의 데이터베이스 관련 문제 대부분을 해결할 수 있습니다.
